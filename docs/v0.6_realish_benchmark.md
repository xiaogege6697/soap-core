# SOAP-Core v0.6.6 Realish Benchmark
日期：2026-06-26

## 1. 目的
验证真实日志接口链路完整：外部风格 CSV（W&B）→ real_adapter 统一 schema → SOAP analyze → taxonomy 对照。并用 normal 数据检验 synthetic taxonomy 是否能迁移到真实接口。零改核心算法、不接 W&B API，仅 CSV。

## 2. adapter field_map 与语义校验
field_map（W&B 风格 → 标准字段）：`train/loss`->`loss`, `eval/loss`->`val_loss`, `train/grad_norm`->`grad_norm`, `lr`->`learning_rate`, `step`->`step`。
`normalize_training_log` 对 `examples/realish_training_wandb.csv`（500 行）映射成功；`describe_fields` 返回 `standard_fields_present=[step,loss,val_loss,grad_norm,learning_rate]`, `missing_optional=[], rows=500`。语义校验通过（`grad_norm`/`loss`/`lr` 全非负）。

## 3. analyze 命令
```bash
PYTHONDONTWRITEBYTECODE=1 .venv/bin/python -m soap.cli analyze examples/normalized_realish_training_wandb.csv --embedding window --method pca --predictor simplex --recurrence --neighbors 12 --recurrence-quantile 0.1 --clusters 3 --output-dir outputs_v066_realish_wandb --report-dir reports_v066_realish_wandb
```

## 4. 关键发现：step schema 陷阱
首次跑 normalized CSV 时 d*=3（异常）。根因：normalized CSV 的 step 列写成数值（0-499），SOAP 的 `load_csv` 把数值首列当成状态空间维度参与 PCA（step 单调递增 = 一个独立维度），导致 d* 虚高。synthetic normal 的 d*=2 是因为其首列是字符串（`step_0`）被 `load_csv` 当 timestamp 跳过。
修复：normalized CSV 的 step 写成字符串（`step_{i}`），让 SOAP 当 timestamp 跳过。修复后 d*=2（正确）。
教训：normalize 输出的标准 CSV，step 必须为 timestamp 字符串，不能是数值，否则 SOAP 会把训练步数当成状态维度。这是真实接口的关键 schema 规则。

## 7. 结论
- **链路完整验证**：外部 W&B 风格 CSV → real_adapter normalize → 标准 schema CSV → SOAP analyze → 结构指纹，全链路打通。
- **taxonomy 迁移成功**：normal 数据在真实接口下呈 normal 指纹（与 synthetic 一致）。
- **关键教训**：normalize 输出的 step 必须为字符串（timestamp 角色），否则 SOAP 把训练步数当维度（d* 虚高）。建议 real_adapter 后续在写标准/临时 CSV 时把 step 输出为字符串。
- **边界**：realish 数据本质仍是合成 normal 数据（仅字段名 W&B 风格），非真实训练日志；真实 run 的 taxonomy 迁移仍待 v0.6.7+ 验证。
