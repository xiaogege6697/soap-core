# SOAP-Core v0.6.5 Real Training Log Schema
日期：2026-06-26

## 1. 目的
本文档旨在定义 SOAP 训练日志（training log）的标准字段（schema）与语义校验规则（semantic validation rules）。此 schema 将作为处理真实日志数据（来自平台如 W&B、TensorBoard 或 HuggingFace Trainer 的 CSV 导出）的统一入口（unified adapter）。核心目标是通过首先统一日志结构，避免因字段语义混乱（例如，`grad_norm` 出现负值）而扭曲后续的 SOAP 指纹计算。本轮实现（v0.6.5）仅支持解析 CSV 格式的导出文件，暂不集成命令行界面（CLI）或平台原生 API（如 W&B API、TensorBoard event files）。

## 2. 标准字段定义
下表列出了经过标准化的日志字段及其规范。所有字段在适配器内部都应使用其英文名进行引用。

| 字段 (Field) | 必需 (Required) | 类型 (Type) | 说明 (Description) |
| :--- | :--- | :--- | :--- |
| `step` 或 `timestamp` | 必需 | `int` / `str` | 训练步数或时间戳，作为数据点排序的基准，应呈现单调递增趋势。 |
| `loss` | 必需 | `float` (≥ 0) | 训练损失值。 |
| `val_loss` | 可选 | `float` (≥ 0) | 验证损失值。 |
| `grad_norm` | 可选 | `float` (≥ 0) | 梯度范数（gradient norm）。如果字段存在，其值必须为非负数。 |
| `learning_rate` | 可选 | `float` (≥ 0) | 当前学习率（learning rate）。 |

## 3. 字段语义校验规则
为确保用于生成 SOAP 指纹（特别是 `d*` 指标）的数据质量，适配器必须对读入的数据执行以下语义校验。校验是数据分析与可视化的前提。

*   **`loss` / `val_loss`**：值必须为非负数（`>= 0`）。若存在负值，适配器应报错（error）并终止该日志的处理。
*   **`grad_norm`**：值必须为非负数（`>= 0`）。若存在负值，应报错。这是 v0.6.4 版本中发现的关键教训：负的梯度范数会严重扭曲 `d*` 的计算。
*   **`learning_rate`**：值必须为非负数（`>= 0`）。若存在负值，应报错。
*   **`step`**：字段值应呈现单调递增（monotonically increasing）的趋势。
    *   若存在轻微的非单调（non-monotonic）情况（例如，因日志记录抖动导致的少量重复或回退），适配器应记录警告（warning）至元数据（metadata）中，但继续处理数据。
    *   仅当该字段呈现**严格递减**（strictly decreasing）的趋势时，才应视为严重错误并报错，这表明数据可能发生了完全错乱。

## 4. 缺失字段回退策略 (Fallback Strategy)
当标准字段在输入的 CSV 中缺失时，适配器应按以下策略处理：

*   **`loss` 缺失**：这是核心分析字段，其缺失将导致日志不可用。适配器应报错（error）。
*   **`step` / `timestamp` 缺失**：适配器可将数据行索引（从 0 开始）用作 `step` 的替代值（即生成 `0, 1, 2, ...`）。
*   **`val_loss` 缺失**：可以继续处理。后续分析流程将不包含 `val_loss` 相关的图表和指标。
*   **`grad_norm` 缺失**：可以继续处理。但请注意，**坍缩型二级分类**（例如，区分 divergence 与 collapse）的算法依赖于梯度信息。若缺少 `grad_norm`，系统只能提供一级分类（如 坍缩 / 扩张 / 基线）。
*   **`learning_rate` 缺失**：可以继续处理。后续分析将不包含学习率相关的图表和指标。

## 5. 外部字段映射 (field_map)
不同的训练框架导出的日志 CSV 使用不同的列名。适配器需要一个可配置的映射表 (`field_map`) 将外部列名映射到上述标准字段。以下是三种典型平台的映射示例：

| 平台 (Platform) | 外部列名 (External Column Name) | → | 标准字段 (Standard Field) |
| :--- | :--- | :--- | :--- |
| **W&B 风格** | `train/loss` | → | `loss` |
| | `eval/loss` | → | `val_loss` |
| | `train/grad_norm` | → | `grad_norm` |
| | `lr` | → | `learning_rate` |
| **TensorBoard scalar CSV** | 通常 `value` 列包含数据，需通过 `tag` 列（如 `train/loss`）进行拆分和映射。映射关系与 W&B 风格相同。 | | |
| **HuggingFace Trainer log** | `loss` | → | `loss` |
| | `eval_loss` | → | `val_loss` |
| | `grad_norm` | → | `grad_norm` |
| | `learning_rate` | → | `learning_rate` |

## 6. 适用范围与限制 (Boundaries)
*   **输入格式**：当前版本（v0.6.5）仅支持解析结构化的 **CSV 导出文件**。暂不支持通过 API 直接获取 W&B runs 数据或解析二进制的 TensorBoard event 文件。这些功能计划在后续版本中实现。
*   **校验深度**：当前语义校验仅执行最小必要集（如非负性、单调性趋势），**不校验数值的合理性**（例如，`loss` 值异常巨大但非负的情况）。
*   **数据质量**：真实世界的数据可能包含 `NaN`（非数字）或 `Inf`（无穷大）值。适配器在预处理阶段应能够识别这些值，并采取策略（如跳过该行或标记为错误），以防止其污染后续计算。

## 7. 后续计划 (Next Steps)
**v0.6.6**：使用真实的或半真实的 CSV 日志数据集进行基准测试（benchmark）。主要目标是验证基于合成数据建立的 SOAP 异常分类法（taxonomy）能否有效迁移到真实场景，并测试当关键字段（如 `grad_norm`）缺失时，系统的判断能力与降级情况是否符合预期。

## 8. v0.6.7 step/timestamp 输出规则固化
- 给 `soap.cli analyze` 使用的 CSV，第一列必须是非数值的 `timestamp`（例如 `step_0`, `step_1`, `step_2`）。
- 如果原始数据或外部适配器提供的 `step` 是纯数字，适配器在输出最终 CSV 时，应将其转换为 `step_{value}` 的字符串格式。
- 原因：`soap.data.loader.load_csv` 会将数值型首列自动识别为特征列，参与后续的 PCA 计算，导致 `d*` 指标异常虚高。实测表明，在 v0.6.6 中，当首列为数字 `step` 时，`d*=3`；而转换为 `step_{value}` 字符串后，`d*=2`，符合预期。
- 为固化此规则，v0.6.7 在 `real_adapter` 中新增了 `write_normalized_training_csv(source_path, output_path, field_map)` 函数。该函数封装了完整的处理流程：调用数据标准化（normalize）和描述（describe），并在输出标准 CSV 时，强制将首列 `step` 字段值转换为 `step_{value}` 字符串。所有提供给 SOAP 工具链的 CSV 文件都应经由此函数产出。
- 已编写单元测试 `test_write_normalized_csv_step_is_string` 以覆盖此规则，确保转换逻辑正确。
