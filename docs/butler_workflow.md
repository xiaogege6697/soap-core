# SOAP-Core Butler Workflow

SOAP-Core 使用“主线程管家 + 临时工 subagent”的协作方式推进。

来源参考：<https://github.com/xiaogege6697/butler-skill>

## 1. 分工原则

主线程只做高价值决策：

- 产品定位
- 核心抽象
- 算法路线选择
- 代码合并前验收
- 用户沟通与阶段总结

临时工负责可并行、可验收的任务：

- 开源项目调研
- 论文/前沿方法调研
- 竞品分析
- 独立模块实现
- 测试补充
- 文档初稿

红线：核心模型、数据结构、维度选择逻辑不能被多个 subagent 拼接式设计，必须由主线程统一把关。

## 2. 派发五要素

每个 subagent 任务必须包含：

1. 目标：要完成什么
2. 输入：给什么资料/路径/上下文
3. 约束：不能做什么、范围边界
4. 验收标准：怎样算完成
5. 超时预期：多久内回来，超时怎样降级

反例：

```text
帮我调研一下复杂系统项目。
```

正确：

```text
目标：找出 10 个与 SOAP-Core 相近的开源项目。
输入：SOAP-Core 定位：有效维度搜索 + 吸引子预测。
约束：优先 GitHub/官方文档/论文，不要博客二手信息。
验收：输出名称、URL、语言、stars、核心能力、可借鉴点。
超时：2 分钟内返回摘要，不完整也要标注缺口。
```

## 3. 检查点

轻任务只走 CP4/CP5；多 agent 重任务走完整检查点。

### CP1：任务拆解检查

派发前确认每个子任务互相独立、目标清楚、输出格式明确。

### CP2：执行端健康检查

派发后确认 subagent 正常启动；连续失败 2 次则换路径或主线程接手。

### CP3：中间产物验收

抽查：

- 输出格式是否符合要求
- 条目数量是否足够
- 关键事实是否能被一手来源支撑

### CP3.5：精度守恒

多步传递必须保留：

- URL 原样
- 文件路径原样
- 数字原样
- JSON/CSV 字段原样
- 算法名/论文名不改写

发现丢精度，回到丢失点重做；无法修复则主线程接手。

### CP4：最终交付检查

合并前检查：

- 子任务结果是否收齐
- 结论是否冲突
- 决策是否有足够证据
- 是否需要用户确认

### CP5：复盘落盘

任务完成后，把重要踩坑、决策和路线写进项目文档。

## 4. SOAP-Core 专用派发模板

### 开源项目调研

```text
目标：调研与 SOAP-Core 相近的开源项目。
输入：SOAP-Core = 领域无关的有效维度搜索 + 相空间重构 + 吸引子预测底座。
约束：优先 GitHub、官方文档、论文；不要只引用博客。
验收：输出项目表（名称、URL、语言、stars、核心能力）、可借鉴点、差异点、优先学习 Top 5。
超时：2-5 分钟。
```

### 论文前沿调研

```text
目标：调研某一方法谱系的学术前沿。
输入：指定主题，如 EDM / Koopman / TDA / Intrinsic Dimension。
约束：优先论文、官方项目、综述。
验收：输出方法谱系、代表论文、对 SOAP-Core 意义、2020 年后新方向。
超时：2-5 分钟。
```

### 独立模块实现

```text
目标：实现某个独立模块。
输入：文件范围、接口约定、已有测试。
约束：只改指定文件；不要重构无关代码；不要覆盖他人改动。
验收：列出改动文件、运行命令、测试结果、已知限制。
超时：5-15 分钟。
```

### 验证/审查

```text
目标：审查某个模块的正确性与边界问题。
输入：目标文件路径、预期行为、测试命令。
约束：不改代码，除非明确要求。
验收：输出问题清单，按严重度排序，给出复现方式和建议修复。
超时：2-5 分钟。
```

## 5. 项目红线

1. 不派模糊任务。
2. 不把核心架构拆给多个 subagent 各写一块。
3. 不跳过主线程验收。
4. 不让 subagent 通过记忆传递关键数据，结构化数据必须落文件。
5. 不改写数字、URL、路径、论文名、算法名。
6. 不无限重试，失败 3 次必须换路径或降级。
7. 不做“又一个 dashboard”，SOAP-Core 定位是状态空间风险解释层。
